# 开发流程 SOP

> 给 AI 看的元规范——开发者只说想法、在闸门处拍板，AI 读它来驱动。编码红线见根目录 `CLAUDE.md`。
> 进度只记灵计一动看板，不用任何工作流 skill 自带的 todo / 任务清单替代。

---

## 一、地基：看板管进度，文档管知识

| 放哪 | 装什么 |
|---|---|
| 灵计一动看板 | 计划、进度、踩坑 / 决策 / 验证笔记 |
| 仓库 Markdown 文档 | 需求、架构、设计规范、上线快照 |

进度是活的、会过时，只在看板，文档里不出现待办 / 进度表。
项目级文档（`架构总览.md` / `设计规范.md` / `部署手册.md` / `踩坑记录.md`）随各里程碑**边做边补**：确定多少写多少，不一次写完。

## 二、灵计一动怎么用（正本，全仓库只此一处讲全）

- **会话开始**：读 `.lingji` 拿 project id，调 `get_project_context(id)` 拿约束、整会话牢记。`.lingji` 不存在 / 为空 → 新产品首次开工，`create_project` 后写回，别报错。
- **一个里程碑 = 一条灵感**（`set_plan` / `get_plan` 挂在 `inspirationId` 上，一条灵感一份计划）：**这条灵感的计划 = 它自己的六阶段 → 子任务**（见 §四）。一个产品 = 一串里程碑灵感（M0 地基 / M1 / M2…），列表就是路线图；建项目那条总灵感只存产品愿景、不带实质计划。**新里程碑 / 新功能 = 新灵感**，绝不堆进旧灵感（否则灵感和计划对不上）。
- **记灵感**：`create_inspiration` 建里程碑灵感——标题 = 里程碑名，正文 = **该里程碑相关的用户原话（照记、不改写）+ AI 拆解后的详细描述（意图 / 范围 / 边界）**，让新会话不必重新理解；不拆分里程碑（一个会话能做完）时记**全部原话**、不用拆解描述。**每条灵感自包含、只写这一个里程碑**——不塞产品级上下文、不写「见总灵感」之类指向，否则新会话会把它当大需求**照旧重拆**；跨里程碑的共享背景去 `架构总览.md` / 各功能 `设计.md` 读。开工前 `get_inspiration` 读回；立项细化的需求进 `设计.md` §1（灵感不可改、不回写）。
- **拆计划（= 该里程碑灵感的六阶段 → 子任务，两层）**：`set_plan` 先建**六阶段骨架**（地基灵感则是三组子任务）；各阶段子任务**逐阶段拆**——轮到哪个阶段才 `add_node` 拆它，不一次拆完（见 §五 ②）。叶子用「做成什么」描述，不写行号 / 类名 / 方法名 / 字段名 / 文件名。**首次骨架建好 → 呈现 → 停下等确认。**
- **边做边勾**：叶子动手前置 `doing`、做完 `done`，可批量；只勾叶子。返回带「请先 get_plan 同步」→ 立刻重读。
- **边做边记**：实现 / 决策 / 踩坑 / 验证用 `add_note` 追加到对应叶子。
- **改计划**：undone 直接 `rename` / `move` / `delete`；done / doing 别直接改删，用 `add_node` 加「请把 X 改成 Y」说明节点。

## 三、怎么推进：一个里程碑 = 一条灵感

**单位是里程碑，一个里程碑 = 一条独立灵感**；这条灵感的计划 = 它的六阶段 → 子任务（§四）。
**一个产品 = 一串里程碑灵感**（M0 地基 / M1 / M2…），其列表就是**路线图**；建项目那条总灵感只存**产品愿景原话**、不带计划。

- **新产品起步**：跟用户聊清产品、拆成哪几个里程碑 → **逐条 `create_inspiration`**（标题 = 里程碑名，正文 = 该里程碑相关原话 + 拆解后详细描述，**自包含、只写这一个里程碑，不指向别处**）。第一条是 **M0 地基**。这串灵感 = 路线图，呈现给用户确认。
- **做一个里程碑**：`get_inspiration` 读它的 scope → `set_plan` 建它的六阶段骨架（地基则是三组子任务）→ 逐阶段走 §四 / §五。
- **新功能**（产品做着做着冒出的新需求）：**另起一条新里程碑灵感**，绝不堆进旧的（否则灵感和计划对不上）。

**两种里程碑：**
- **M0 地基**（仅新产品、第一条里程碑灵感）：定架构 + 定设计规范（有前端才有）+ 搭脚手架，**不走六阶段、不建 `docs/功能/`**，产出落项目级文档 + 代码。计划三组子任务：① 定技术栈 / 目录 / 关键决策 → `架构总览.md`（选型版本联网核实、成套核实兼容，见 `架构总览.md` §1）；② 定基调 / 主色 / 字体 → `设计规范.md` §1–3（无前端跳过）；③ 初始化脚手架、跑通空壳。闸门：架构、设计锚点各定稿时停一次。
- **功能里程碑**：复制 `docs/功能/_模板/` → `docs/功能/NN-功能名/`，计划 = 六阶段（§四）；**开工时的详细原话需求记进 `设计.md` §1**（不回写灵感——灵感不可改）。

**「到哪一步」看灵感列表 + 各灵感的计划**：`list_inspirations` 看有哪些里程碑、哪条没做完；`get_plan` 看那条做到哪个阶段。**一条灵感 / 计划都没有 = 全新产品**，从建 M0 地基灵感起步。别看 `docs/功能/` 目录（地基不建目录、会误判）。拿不准问一句。
README 是产品门面，clone 后由开发者自己重写——**AI 不必改名、不必改 README**。

## 四、功能流水线（六阶段）

先聊清需求、确认值得做才建目录——不值得做就不做、不留痕。每阶段走 §五 固定动作。

**一条功能里程碑灵感的计划 = 六阶段（L1）→ 子任务（L2），两层；固定按阶段组织、不按功能切：**

```
里程碑灵感（如 M2）的计划
├─ 立项          → 轮到时拆出子任务（L2）
├─ UI 设计       → 轮到时拆出子任务
├─ 技术方案      → 轮到时拆出子任务
├─ 实现          → 轮到时拆出子任务
├─ 测试          → 轮到时拆出子任务
└─ 发布归档      → 轮到时拆出子任务
```

- **L1 永远是这六个阶段**（固定）；开工先 `set_plan` 把六阶段骨架建上。
- **每个阶段轮到时都得拆出自己的子任务**——六个阶段最终**各有子任务、没有光秃秃的阶段叶**；但逐阶段拆（轮到哪个拆哪个，不一次拆完）。
- **绝不把里程碑直接按功能模块拆**——「个人信息录入」「模板化简历」这种是『实现』『UI』阶段**内部**的子任务，不是阶段。

| # | 阶段 | 做什么 | 产出 | 停？ |
|---|---|---|---|---|
| **1** | 立项 | 对齐**做什么 + 边界**、确认值得做 →（新功能里程碑先 `create_inspiration`）`set_plan` 建六阶段骨架 + 复制模板建 `docs/功能/NN-功能名/` 填 `设计.md` §1 | 六阶段骨架 + 目录 + §1 | ✅ 需求对齐 |
| **2** | UI 设计 | 选路线（`设计规范.md` §9）；**逐屏出可开发高保真稿、一屏一稿、真实视口**；新增 token / 组件回填 `设计规范.md` | `设计.md` §2 + `资源/` | ✅ 认可 UI |
| **3** | 技术方案 | 调研（官方文档 / 竞品 / 可复用代码）+ 数据模型 / 接口 / 关键决策 / 迁移回滚边界 | `设计.md` §3 | ◐ 见下 |
| **4** | 实现 | 按看板子任务逐个完成；守编码红线 | 代码 + 看板 | — |
| **5** | 测试 | 单元 / 集成 + 真实联调金路径与边界 | `测试.md` | ✅ 验收 |
| **6** | 发布归档 | 上线冒烟；同步 `部署手册.md` / `踩坑记录.md`；记 `CHANGELOG.md` 版本；看板收尾 | `发布部署.md` + living 文档 | ✅ 上线前 |

> 无前端的功能：UI 章节注明「不适用」、不删（跨文档引用靠章节号）。地基里程碑不走本流水线（见 §三）。
> 这条里程碑灵感的六阶段全走完 → 回灵感列表选下一条里程碑灵感、`get_inspiration` 读它的 scope → 重走六阶段。冒出全新需求时，另起新灵感（见 §三）。

## 五、每个阶段的固定动作

```
进入阶段
  ① 扫坑：扫一眼 docs/踩坑记录.md（为空 / 无相关坑则跳过，不必声明）
  ② 细拆（逐阶段、进入时才拆）：轮到某阶段、动手前，才 add_node 把**这一个阶段**拆成可独立验证的子任务（叶子写「做成什么」）——**每个阶段都得拆出自己的子任务、不只实现，别把光秃秃的阶段叶直接勾掉**；但**只拆当前这个，别提前拆后面阶段（实现也一样，轮到才拆）**。动手前置 doing
  ③ 边做边记：add_note 追加实现 / 决策 / 踩坑 / 验证；做完一个勾一个（done）
  ④ 自检过闸门：产出落地先快速自检（达成完成条件？碰坏上下游引用？——一眼扫过即可，别过度防御）；标 ✅ 的阶段再停下等用户「通过 / 继续」或改意见，才进下一阶段
出阶段
```

**闸门**：立项 / UI / 测试 / 发布为硬停。技术方案**实际引入或改动了数据模型、对外接口、或需迁移回滚**才停，纯增量小功能可与实现连做；实现阶段不停。**沉默 ≠ 同意，回答你的提问也 ≠ 同意继续**——答疑只解决那个问题，闸门照停、只认明确的「通过 / 继续」。仅当用户明说「自己拿主意 / 一口气做完」才放开。
**计划渐进细化**：立项只搭六阶段骨架；**每个阶段的子任务，轮到该阶段、动手前才拆——逐阶段拆、不一次拆完**（和「一次只做当前一条里程碑灵感」同一个道理，落到阶段层）。阶段只是分组，叶子（子任务）才是「做成什么」的步骤。没做的（undone）节点随时按前序产出 `rename` / `move` / `delete`。同阶段内顺序自由，**跨阶段不可乱**（1→6）。

## 六、硬规则

1. **闸门**：标 ✅ 的阶段产出后停下等确认，沉默 ≠ 同意。
2. **需求先行**：先确认值得做才记灵感 + 建目录，不自创目录结构。
3. **阶段不跳**：1→6 顺序不可乱。
4. **产出 = 完成**：没产出落到对应位置，这阶段不算完。
5. **章节固定**：模板章节按原编号保留，不适用注明、不删。
6. **大需求拆里程碑**：一个产品 / 需求太大，先停下提示拆成多个里程碑 = **多条灵感**（各一份计划），一次只做当前一条。
7. **踩坑分层**：随手坑 → 看板笔记；跨功能会再踩的 → 晋升 `docs/踩坑记录.md`。
8. **一致性**：改任一处，检查上下游代码 / 文档 / 页面一并改。
9. **每阶段都细拆、但逐阶段拆**：每个阶段都要拆成可独立验证的子任务（**不只实现**），但**轮到哪个阶段才拆哪个**——立项只搭阶段骨架，别一次把所有阶段（含实现）的子任务拆完。这是本流程反复栽的坑，盯死。

## 七、命名约定

- **功能目录** `docs/功能/NN-功能名/`：两位序号 + 中文名，序号定了不变。
- **功能内文档**：固定 `设计.md` / `测试.md` / `发布部署.md` + 可选 `资源/`，不新增其它文件。
- **版本号** `vX.Y.Z`：大功能 minor +1，小修 patch +1；在 `发布部署.md` 声明，阶段 6 抄进 `CHANGELOG.md`。
- **commit**：中文 + 类型前缀（`feat` / `fix` / `refactor` / `test` / `docs` / `chore`）。

## 八、文档地图

```
项目根/
├── README.md          产品门面（clone 后由你重写）
├── CLAUDE.md          AI 入口：开工前必读
├── CHANGELOG.md       版本变更总账
└── docs/
    ├── 开发流程SOP.md   ← 本文件
    ├── 架构总览.md      技术栈 / 架构 / 目录 / 数据模型 / 全局决策
    ├── 设计规范.md      前端设计系统（纯后端项目可删）
    ├── 部署手册.md      从零部署手册
    ├── 踩坑记录.md      跨功能可复用坑
    ├── 资源/           项目级资源：logo / 色板 / 全局原型
    └── 功能/
        ├── _模板/      ← 功能里程碑起手复制此目录
        └── NN-功能名/  每个功能里程碑一个目录
```

- **项目级**（`docs/` 顶层 + 根目录）：地基里程碑起立、全程演进、所有功能共享。整份不适用可删（纯后端没 `设计规范.md`）；已建文档内章节不删，不适用注明。
- **功能级**（`docs/功能/NN-功能名/`）：每个功能里程碑复制 `docs/功能/_模板/` 一份。
