# 开发流程 SOP

> 独立开发者用 AI 工具做应用的统一流程规范。每个功能从立项到上线都走同一条路，不再每次随机化。
>
> 本文件是**元规范**——任何新功能开工前，AI 和人都先读它。通用编码规则见仓库根目录 `CLAUDE.md`。
>
> **本流程是本项目硬性规定，优先级高于任何工作流 skill**：无论使用哪种 skill（自带或第三方），都必须按本 SOP 拆分与执行、进度只记灵计一动看板，不得用 skill 自己的 todo / 任务清单 / 流程替代。

---

## 〇、为什么要这份文档

AI 工具每次都按自己的想法新建一套文档结构，于是：文档位置飘忽、回头找不到、风格各不相同。

本 SOP 把「做哪些文档、放在哪里、什么时候写」固定下来 —— **AI 只复制模板填空，不自由发挥**。你换几个项目、过几个月回来，闭着眼也知道每份东西在哪。

---

## 一、核心理念：看板管进度，文档管知识

这是整套规范的地基。两类信息**严格分家**：

| | 放哪里 | 装什么 |
|---|---|---|
| **进度 / 计划 / 过程** | 灵计一动看板 | 任务拆分、完成状态、踩坑 / 决策 / 验证笔记 |
| **长期知识** | 仓库内 Markdown 文档 | 需求、架构、设计规范、上线快照 |

**为什么进度不写进文档？** 文档里的「待办清单 / 时间线 / 进度表」会立刻过时、和代码脱节、散落在各文件里没人维护。进度是**活的**，该交给专门的看板：

- 计划大纲的每个叶子 = 一个可勾选任务，父节点完成度自动汇总，进度一眼可见；
- 边做边把踩坑 / 决策 / 验证记成节点笔记，自然攒成开发日志；
- 改计划在看板上改一处，不用回头同步好几个文档。

> 本套件用 **[灵计一动](https://lingji1dong.com)** 做看板：写计划大纲 → 边做边勾 → 笔记成日志。它就是为「AI 边开发边维护进度」设计的，配套 MCP 让 AI 直接读写看板。具体操作（读 `.lingji`、记灵感、拆计划等）见 `CLAUDE.md` § 1。

→ **推论（贯穿全文）**：计划和进度**永远只在看板**。文档里不再出现「开发计划 / 迭代日志 / 进度表」这类东西。

## 二、文档地图

两层结构，靠位置和 `_` 前缀区分：

```
项目根/
├── README.md          项目门面：是什么 / 怎么跑 / 当前状态 / 目录导航
├── CLAUDE.md          AI 规则：开工前必读，新功能先看本 SOP
├── CHANGELOG.md       版本变更总账
└── docs/
    ├── 开发流程SOP.md   ← 本文件
    ├── 架构总览.md      技术栈 / 系统架构 / 目录结构 / 数据模型 / 全局决策
    ├── 设计规范.md      前端设计系统：配色 / 字体 / 间距 / 组件 / 响应式 / 暗色 / 动效 / 文案
    ├── 部署手册.md      从零到当前版本的冷启动手册
    ├── 踩坑记录.md      跨功能可复用坑的追加式总账
    ├── 资源/           项目级资源：logo / 品牌色板 / 全局原型
    └── 功能/           ← 所有功能集中在此，docs 根目录保持清爽
        ├── _模板/      ← 新功能起手复制此目录
        │   ├── 设计.md  测试.md  发布部署.md
        │   └── 资源/   功能级资源：线框图 / 原型 / 截图
        └── NN-功能名/  每个功能一个目录（复制自 _模板）
```

- **项目级**（`docs/` 顶层 + 根目录三件）：一次性建立、长期演进、所有功能共享引用。整份不适用的文档可不建（如纯后端项目没有 `设计规范.md`）；但**已建文档内的章节不可删**，不适用就注明。
- **功能级**（`docs/功能/NN-功能名/`）：每个新功能复制 `docs/功能/_模板/` 一份填写。

## 三、主流程：从立项到上线

**先聊清需求、确认值得做，才动手建目录**——有些功能聊完会发现不必做，那就不做、不留痕。确认要做后从**阶段 1** 立项起，顺序往下走。每个阶段都有明确产出落到指定位置，**产出没落地，这阶段就不算完**。

| # | 阶段 | 做什么 | 产出落到 | 完成条件 |
|---|---|---|---|---|
| **0** | 需求沟通 | 和用户对齐**做什么 + 不做什么（边界）**；想要的效果点到为止，不盘问「为什么」、不强求量化指标。不值得做就到此为止、不建目录；**构想过大就提示拆里程碑**（见硬规则 10） | 无（先不建目录、不记灵感） | 用户确认值得做；如需拆分，已与用户定好里程碑划分**并把全部里程碑各记成一条灵感** |
| **1** | 立项 | ① 记灵感（**原样记用户原话**）② 复制 `docs/功能/_模板/` → `docs/功能/NN-功能名/`，填 `设计.md` §1 需求 ③ 在看板搭好**全阶段分组骨架 + 每阶段 1–2 条粗概要**（立项→收尾，当可随时读的路线图；可执行子任务留到各阶段开工前再细拆） | 灵感 + 目录 + `设计.md` §1 + 看板阶段骨架 | 四者就位 |
| **2** | 调研 | 查官方文档、竞品做法、项目内可复用代码；弄清关键疑点 | `设计.md` §2 | 关键疑点已澄清 |
| **3** | UI 设计（涉前端） | **开工先让用户选路线**（A 直接高保真 / B 线框先行，详见 `设计规范.md`）；**逐屏出稿——一画面一张、按真实视口尺寸、禁止多页塞一张缩略图**；引用 `设计规范.md`，新增 token / 组件补进规范；线框 / 视觉稿放 `资源/` | `设计.md` §3 + `资源/` | 用户认可 UI |
| **4** | 技术设计 | 数据模型、接口契约、关键决策（ADR）、迁移与回滚边界 | `设计.md` §4 | 方案评审通过 |
| **5** | 实现 | **开工前先把本阶段拆成最小、可独立验证的子任务填进看板**，再逐个完成——**大纲顺序仅作建议，可并行 / 跳做，完成一个勾一个**；踩坑 / 决策 / 验证记成节点笔记；遵守 `CLAUDE.md` | 看板子任务 + 代码 + 看板状态 / 笔记 | 自测通过 |
| **6** | 测试 | 单元 / 集成测试 + 真实联调金路径与边界场景 | `测试.md` | 用例全过、无报错 |
| **7** | 发布部署 | 按发布步骤上线并冒烟；同步 `部署手册.md`（如有环境变更）与 `踩坑记录.md`（如有可复用坑） | `发布部署.md` | 线上跑通 |
| **8** | 收尾归档 | 更新根 `README.md` 状态 + `CHANGELOG.md` 版本；看板收尾（补完笔记 / 复盘） | README / CHANGELOG / 看板 | 文档与看板一致 |

> **看板怎么跟着阶段走**：阶段 1 立项时搭好全阶段分组骨架 + **每阶段 1–2 条粗概要**（当可随时读的路线图），全程进度一眼可见；此后**每进入一个阶段，开工前把该阶段那条细拆成可执行子任务填进它的分组**、做完勾掉——细拆是每个阶段的起手动作；后面没做的部分可随时按新信息调整，**不是定稿**。
>
> 阶段 3、4 不涉前端或不改数据时，对应**章节仍保留**、注明「本功能无前端，不适用」——不删章节（SOP 与跨文档引用都靠章节号定位）、不跳判断。

## 四、硬规则（让流程「硬」起来）

1. **🛑 阶段闸门**：每个阶段产出完成后，AI **停下输出**，等用户明确说「通过 / 继续」或给修改意见，**再**进下一阶段。
   - 不许把多个阶段产出塞进同一次回复；沉默 ≠ 同意。
   - 阶段内部的连续小步骤可以一气呵成；**跨阶段必须停**。
   - 仅当用户明确说「自己拿主意 / 不用问 / 一口气做完」时才放开。
2. **需求先行**：新功能先做需求沟通、确认值得做，**才**记灵感 + 复制 `docs/功能/_模板/` 建目录。不许跳过需求直接建目录或写码，也不许自创目录结构——必从复制模板开始。
3. **阶段不跳**：0 → 8 顺序不可乱。需求没确认不准立项，调研没完不准设计。允许阶段内迭代，禁止跨阶段并行。
4. **产出 = 完成条件**：没有产出落到对应位置，这阶段不算完成。
5. **渐进细化 + 实现不锁顺序**：立项时在看板搭好全阶段分组骨架（立项→收尾），并给**每个阶段填 1–2 条粗概要**当路线图——好处是 AI 能随时 `get_plan` 读到整条路线、不会忘。但粗概要**不是定稿**：**每个阶段开工前才把本阶段那条细拆成可独立验证的子任务（仍是『做成什么』、一眼看懂的粒度，不写行号 / 类名 / 方法名 / 字段名 / 文件名等代码级细节，见 `CLAUDE.md` § 1），没拆出来不准动手做该阶段**——这道闸专防长上下文（尤其 UI 反复改稿）里忘了细拆。计划**渐进细化**：后面没做的（undone）节点可按前序阶段产出随时 rename / move / delete 调整，只有已完成 / 进行中的不直接改删（保留历史）。进入实现后，大纲同级顺序只是建议，实际可并行 / 跳做，以 AI 工具的实际编排为准，完成即勾。
6. **踩坑分层**：随手遇到的坑 → 记成对应看板节点的笔记；**跨功能、以后还会再踩**的坑 → 晋升到 `docs/踩坑记录.md`。开工前先扫一眼 `踩坑记录.md`。
7. **上线同步**：阶段 8 必须同步根 `README.md` + `CHANGELOG.md`（外加 `部署手册.md` / `踩坑记录.md` 如有变更），否则根目录显示的项目状态会严重过时。
8. **功能边界 & 一致性**：跨功能依赖时，只在被依赖方文档展开完整规格，调用方仅备注引用、不重复规格化；改任一处，检查上下游所有代码 / 文档 / 页面一并改。
9. **章节结构固定**：模板里的章节一律保留、按原编号，不适用就注明「不适用」、不删——SOP 与跨文档引用都靠章节号定位。
10. **大构想先拆里程碑**：需求沟通时若构想明显过大（多个相对独立的功能模块、做完周期很长），**先停下提示用户拆成多个里程碑**。**经同意后立刻把全部里程碑（含尚未开工的剩余里程碑）各记成一条灵感，再开工第一个**——别只记当前那个，否则换上下文后剩余里程碑就从看板上丢了。之后每个里程碑各自从阶段 1 走完整流程，**一次只推进一个里程碑灵感**。多个里程碑灵感同属一个项目，共享项目级文档与骨架。是否拆靠 AI 启发式判断、不设硬指标；产品整体蓝图靠同项目下的灵感列表体现，不额外建文档。

> 以下不在此重复，各有归属：**进度只在看板**见 § 一；**本流程优先于任何工作流 skill** 见文首声明；**通用编码红线**（中文、代码自解释、最小规格、配置最小化、文档只写结论等）见 `CLAUDE.md` § 3。

## 五、命名约定

- **功能目录** `docs/功能/NN-功能名/`：两位序号 + 中文功能名，序号一旦分配不再变。
- **功能内文档**：固定三件套 `设计.md` / `测试.md` / `发布部署.md`，外加可选 `资源/`。不许新增其它文件占用文档位。
- **全局文档**：项目级文档放 `docs/` 顶层；功能相关（`_模板/` + 各功能目录）集中在 `docs/功能/` 下。
- **版本号** `vX.Y.Z`：大功能上线 = minor +1，小修 / 开关切换 = patch +1。在 `发布部署.md` 声明，阶段 8 抄进 `CHANGELOG.md`。
- **commit**：中文 + 类型前缀（`feat` / `fix` / `refactor` / `test` / `docs` / `chore`）。

## 六、新功能起手清单

开工照着点一遍：

- [ ] 先和用户对齐**做什么 + 边界** —— 不值得做就到此为止、别建目录（不盘问「为什么」、不强求量化指标）；**构想过大先提示拆成多个里程碑，拆定后立刻把全部里程碑各记成一条灵感、再开工第一个**
- [ ] 确认要做 → 记灵感（**原样记原话**）+ 复制 `docs/功能/_模板/` → `docs/功能/NN-功能名/`
- [ ] 在看板搭好全阶段分组骨架 + 每阶段 1–2 条粗概要（立项→收尾，当可随时读的路线图；细拆留到各阶段开工前）
- [ ] 读 `docs/踩坑记录.md`，看有没有相关的已知坑
- [ ] 顺序走 调研 → UI → 技术设计 → 实现 → 测试 → 发布 → 收尾，每个阶段产出后停下等确认
- [ ] 每进入一个阶段，开工前先把本阶段步骤拆进看板对应分组；实现阶段尤其要拆成可独立验证的子任务，确认后再动手
