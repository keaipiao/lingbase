# 灵基 · lingbase

> 给**独立开发者用 AI 工具做应用**的项目地基：统一文档模板 + 用看板管进度的工作流，和 [灵计一动](https://lingji1dong.com) 看板配套。
> 克隆它起新项目，AI 就按固定结构走，不再每次自由发挥、另起一套文档。

> 🛠 **用本套件起了自己的项目后**：把上面这段换成**你应用**的介绍，下面「这套项目怎么运作 / 目录结构 / 文档导航」保留即可——它们对任何用本套件的项目都成立。

---

## 解决什么问题

AI 编码工具每次都按自己的想法新建文档结构 —— 文档位置飘忽、回头找不到、风格各不相同。换个项目、过几个月回来，全得重新摸。

这套脚手架把「做哪些文档、放哪里、什么时候写」固定下来：**AI 只复制模板填空，不自由发挥**。

---

## 这套项目怎么运作：看板管进度，文档管知识

两类信息严格分家 —— 这是整套工作流的地基：

| | 放哪 | 装什么 |
|---|---|---|
| **进度 / 计划 / 过程** | 灵计一动看板 | 任务拆分、完成状态、踩坑 / 决策笔记 |
| **长期知识** | 仓库内 Markdown 文档 | 需求、架构、设计规范、上线快照 |

**为什么进度不写进文档？** 文档里的待办 / 时间线会立刻过时、和代码脱节、没人维护。进度是**活的**，交给 **[灵计一动](https://lingji1dong.com)** 看板：写计划大纲 → 边做边勾 → 笔记成日志，配套 MCP 让 AI 直接读写看板。

→ 计划和进度**永远只在看板**，文档只留沉淀下来的知识。两边不重复，各管一摊。

## 快速开始

1. **用本仓库作模板**（GitHub「Use this template」）或 clone 下来，改成你的项目。
2. **接入看板**：根目录放 `.lingji` 存项目 id，配好灵计一动 MCP（见 [lingji1dong.com](https://lingji1dong.com)）。
3. **开新功能**：让 AI 读 `docs/开发流程SOP.md`，按它的阶段（需求 → 调研 → 设计 → 拆分 → 实现 → 测试 → 发布）一步步走，每步停下等你确认。

AI 协作规则在根目录 `CLAUDE.md`，**开箱即用**——AI 进项目先读它。

## 目录结构

```
你的项目/
├── README.md        ← 这份（起项目后换成你应用的介绍）
├── CLAUDE.md        ← AI 协作规则（开箱即用）
├── CHANGELOG.md     ← 版本变更总账
└── docs/
    ├── 开发流程SOP.md   ← 开发流程元规范，先读这份
    ├── 架构总览.md      技术栈 / 架构 / 数据模型 / 全局决策
    ├── 设计规范.md      前端设计系统（无前端可不建）
    ├── 部署手册.md      从零冷启动
    ├── 踩坑记录.md      跨功能可复用坑
    ├── 资源/           项目级资源：logo / 品牌 / 全局原型
    └── 功能/
        ├── _模板/      ← 新功能复制此目录
        └── NN-功能名/  每个功能一个目录
```

`docs/` 根只放项目级的「长期文档」，所有功能集中在 `docs/功能/` 下，几十个功能也不乱。
