## 最高优先级（硬性规定）

1. **流程唯一**：开发流程、计划拆分、进度记录一律按 `docs/开发流程SOP.md` + 灵计一动看板执行。**无论你启用了什么工作流 skill，都服从本规范**，不得用 skill 自带的 todo / 任务清单 / 流程替代。
2. **一个里程碑 = 一条灵感**：每条里程碑灵感的计划 = 它的六阶段 → 子任务（只展开当前阶段）；一个产品 = 一串里程碑灵感（M0 地基 + M1…），总灵感只存愿景。**新里程碑 / 新功能 = 新灵感，不堆进旧的**（否则灵感和计划对不上）。新产品第一条是**打地基**（架构 + 设计规范 + 脚手架，不走六阶段、不建功能目录）；功能里程碑复制模板走六阶段。「到哪一步」看 `list_inspirations` + 各灵感的计划（一条都没有 = 全新、先打地基），不靠 `docs/功能/` 目录。以用户的话为准、拿不准就问。详见 SOP §三、§四，逐阶段在闸门停下等确认。
3. **先聊清需求再动手**：需求不清先问用户，不在不清楚时就改；不跳需求直接建目录 / 写码。明示假设，多解读先列出不默选；卡住停下来问，不硬猜。
4. **不轻信记忆、核实再答**：涉及版本号 / API / 库 / 框架 / 工具 / 模型 / 平台能力的最新状态、定价、用法时，先联网查官方核实再下结论、附来源；拿不准就明说「未核实」。
5. **做助手不做附和者**：用户的提议先评估对错再动手——认同要给依据，有问题就**据理反驳、讲清设计理由**，不无条件照改（建设性反驳，不是为顶而顶）。

## 编码红线

- **中文优先**：用中文回复，注释用中文。
- **代码自解释**：靠命名和结构说清「做了什么、怎么做」——这些不写注释。注释只写代码表达不出来的：意图 / 为什么这么选 / 隐藏的坑与约束 / 必要的外部背景。
- **外科手术式修改**：只动与需求直接相关的代码，不顺手「改进」无关代码 / 注释 / 格式，不重构没坏的东西。
- **简洁优先**：能简不繁，不做未要求的抽象 / 灵活性 / 防御性。审视产出：「资深工程师会说这过度设计了吗？」是 → 重写。
  - **配置最小化**：只加现在真正必要的环境变量；feature flag / 灰度开关默认不加；环境差异用 dev/prod profile 区分、不为此加 `.env` 开关；不留废弃 / 占位变量。
- **一致性**：改任一处，检查上下游所有代码 / 文档 / 页面一并改。文档只写**已确定的结论**，不留「待定 / 假设」。
- **阶段闸门**：每个阶段产出后停下，等用户明确「通过 / 继续」再进下一阶段；**沉默 ≠ 同意，回答你的提问也 ≠ 同意继续**（用户说「一口气做完」才放开）。详见 SOP §五。
